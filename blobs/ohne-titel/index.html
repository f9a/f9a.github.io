Eine Datenansicht einer Architektur zeigt den Anwendungszustand an, wenn Informationen durch die Komponenten fließen. Da REST speziell auf verteilte Informationssysteme ausgerichtet ist, betrachtet es eine Anwendung als eine zusammenhängende Struktur von Informations- und Kontrollalternativen, durch die ein Benutzer eine gewünschte Aufgabe ausführen kann. So ist beispielsweise das Nachschlagen eines Wortes in einem Online-Wörterbuch eine Anwendung, ebenso wie das Durchsuchen eines virtuellen Museums oder das Durchlesen einer Reihe von Notizen, um für eine Prüfung zu lernen. Jede Anwendung definiert Ziele für das zugrunde liegende System, an denen die Leistung des Systems gemessen werden kann.

Komponenteninteraktionen erfolgen in Form von dynamisch dimensionierten Nachrichten. Kleine oder mittelgroße Nachrichten werden für die Steuersemantik verwendet, aber der Großteil der Anwendungsarbeit wird durch großformatige Nachrichten geleistet, die eine vollständige Ressourcendarstellung enthalten. Die häufigste Form der Requestsemantik ist das Abrufen einer Repräsentation einer Ressource (z.B. die “GET”-Methode in HTTP), die oft für eine spätere Wiederverwendung zwischengespeichert werden kann.

REST konzentriert den gesamten Kontrollzustand auf die Darstellungen, die als Reaktion auf Wechselwirkungen empfangen werden. Ziel ist es, die Serverskalierbarkeit zu verbessern, indem es entfällt, dass der Server über die aktuelle Anforderung hinaus ein Bewusstsein für den Client-Zustand aufrechterhalten muss. Der Zustand einer Anwendung wird daher durch ihre ausstehenden Anforderungen, die Topologie der verbundenen Komponenten (von denen einige gepufferte Daten filtern können), die aktiven Anforderungen an diese Konnektoren, den Datenfluss von Darstellungen als Reaktion auf diese Anforderungen und die Verarbeitung dieser Darstellungen, wie sie vom Benutzeragenten empfangen werden, definiert.

Eine Anwendung erreicht einen stationären Zustand, wenn sie keine offenen Anforderungen hat, d.h. sie hat keine offenen Anforderungen und alle Antworten auf ihre aktuellen Anforderungen sind vollständig empfangen oder empfangen worden, bis sie als Darstellungsdatenstrom behandelt werden können. Für eine Browseranwendung entspricht dieser Zustand einer “Webseite”, die die primäre Darstellung und zusätzliche Darstellungen wie Inline-Bilder, eingebettete Applets und Style Sheets enthält. Die Bedeutung von stationären Anwendungszuständen zeigt sich in ihren Auswirkungen sowohl auf die vom Benutzer wahrgenommene Leistung als auch auf die Burstiness des Netzwerkanforderungsverkehrs.

Die vom Benutzer wahrgenommene Leistung einer Browseranwendung wird durch die Latenzzeit zwischen den stationären Zuständen bestimmt: die Zeitspanne zwischen der Auswahl eines Hypermedia-Links auf einer Webseite und dem Zeitpunkt, zu dem brauchbare Informationen für die nächste Webseite bereitgestellt wurden. Die Optimierung der Browserleistung konzentriert sich daher auf die Reduzierung dieser Kommunikationslatenz.

Da REST-basierte Architekturen in erster Linie durch die Übertragung von Repräsentationen von Ressourcen kommunizieren, kann die Latenz sowohl durch das Design der Kommunikationsprotokolle als auch durch das Design der Darstellungsdatenformate beeinflusst werden. Die Fähigkeit, die Antwortdaten beim Empfang schrittweise wiederzugeben, wird durch das Design des Medientyps und die Verfügbarkeit von Layoutinformationen (visuelle Abmessungen von Inline-Objekten) innerhalb jeder Darstellung bestimmt.

Eine interessante Beobachtung ist, dass die effizienteste Netzwerkanforderung eine ist, die das Netzwerk nicht nutzt. Mit anderen Worten, die Möglichkeit, eine zwischengespeicherte Antwort wiederzuverwenden, führt zu einer erheblichen Verbesserung der Anwendungsleistung. Obwohl die Verwendung eines Caches aufgrund des Lookup-Overheads jedem einzelnen Request eine gewisse Latenzzeit hinzufügt, wird die durchschnittliche Request-Latenzzeit deutlich reduziert, wenn bereits ein kleiner Prozentsatz der Requests zu brauchbaren Cache-Treffern führt.

Der nächste Steuerzustand einer Anwendung liegt in der Darstellung der ersten angeforderten Ressource, so dass das Erhalten dieser ersten Darstellung eine Priorität ist. Die REST-Interaktion wird daher durch Protokolle verbessert, die “zuerst reagieren und später denken”. Mit anderen Worten, ein Protokoll, das mehrere Interaktionen pro Benutzeraktion erfordert, um Dinge wie die Aushandlung von Feature-Fähigkeiten vor dem Senden einer Inhaltsantwort zu tun, wird wahrnehmbar langsamer sein als ein Protokoll, das das das, was am ehesten optimal ist, zuerst sendet und dann eine Liste von Alternativen für den Client bereitstellt, die er abrufen kann, wenn die erste Antwort unbefriedigend ist.

Der Anwendungszustand wird vom Benutzeragenten gesteuert und gespeichert und kann aus Darstellungen mehrerer Server bestehen. Neben der Befreiung des Servers von den Skalierbarkeitsproblemen der Speicherung von Zuständen ermöglicht dies dem Benutzer, den Zustand direkt zu manipulieren (z.B. die Historie eines Webbrowsers), Änderungen an diesem Zustand zu antizipieren (z.B. Link Maps und Prefetching von Darstellungen) und von einer Anwendung zur anderen zu springen (z.B. Lesezeichen und URI-Eintragdialoge).

Die Modellanwendung ist daher eine Maschine, die von einem Zustand zum nächsten übergeht, indem sie die alternativen Zustandsübergänge im aktuellen Satz von Darstellungen untersucht und auswählt.
